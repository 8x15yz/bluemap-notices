# 웹+Slack 기반 나라장터 공고 추천 MVP PRD

## 1. 개요

블루맵(주)이 수주 가능성이 있는 나라장터 입찰공고를 자동으로 수집하고, 회사 역량 키워드에 맞는 후보를 웹에서 검토할 수 있게 만든다. 새 후보가 생기면 Slack으로 알림을 보내고, 운영자는 웹 상세 페이지에서 원문 공고와 적합도 근거를 확인한 뒤 첨부파일을 업로드해 AI 전략 메모를 받을 수 있다.

## 2. 사용자와 성공 기준

- 주 사용자는 개발자가 아닌 사업/운영 담당자다.
- 사용자는 매일 나라장터를 직접 검색하지 않고도 새 후보 공고를 Slack에서 확인한다.
- 사용자는 Slack 알림의 URL을 눌러 웹 상세 페이지로 이동한다.
- 사용자는 상세 페이지에서 공고 URL, 기관, 마감일, 예산, 키워드 매칭, 적합도 점수를 빠르게 확인한다.
- 사용자는 PDF/HWPX/HWP 첨부파일을 업로드해 추천 이유, 블루맵 역량 매핑, 제안 전략, 리스크, 다음 행동을 담은 전략 메모를 받는다.

## 3. MVP 범위

- 1차 공고 수집원은 나라장터 공공데이터개방표준서비스 `getDataSetOpnStdBidPblancInfo`를 사용한다.
- 2차 상세 보강 수집원은 나라장터 입찰공고정보서비스를 후속 단계로 추가한다.
- 2차 상세 보강은 모든 공고가 아니라 1차 수집에서 키워드/점수 기준으로 후보가 된 공고만 대상으로 한다.
- MVP 초기 구현은 1차 공공데이터개방표준서비스만 동작하며, 입찰공고정보서비스는 환경변수와 설계 기준을 먼저 준비한다.
- 후속 수집원 후보는 MSIT, 기업마당, e나라도움, K-Startup, IRIS로 기록하되 구현하지 않는다.
- Slack은 새 후보 공고 알림만 담당한다. Slack slash command, Slack 파일 업로드 분석, 양방향 상태 변경은 후속 기능이다.
- 첨부파일 분석은 웹 상세 페이지에서만 제공한다.
- DB는 Postgres를 사용한다.

## 4. 핵심 기능

- 정해진 주기로 나라장터 입찰공고를 조회한다.
- 마감일이 지난 공고는 후보에서 제외한다.
- 아래 키워드 중 하나 이상이 공고명, 기관명, 업종명, 업무구분, 지역, 기타 원문 텍스트에 포함되면 후보로 저장한다.
- 키워드는 동등 가중치로 계산한다.
- 이미 Slack 알림을 보낸 공고는 중복 발송하지 않는다.
- 낙찰방법이 `수의시담`인 공고는 1차 목록 응답에서 확인되는 즉시 후보 저장, 상세 보강, Slack 알림 대상에서 제외한다.
- 웹 목록은 적합도 점수와 마감일 기준으로 우선순위가 보이게 정렬한다.
- 웹 상세 페이지는 첨부파일 업로드와 AI 전략 메모 생성을 제공한다.
- 후속 상세 보강 단계에서는 후보 공고의 입찰공고번호/차수를 기준으로 입찰공고정보서비스를 조회한다.
- 입찰공고정보서비스에서 받은 첨부파일 URL, 공고규격파일명, 참가 가능 지역, 투찰 가능 업종, 계약 방식, 기초금액 등은 웹 상세 페이지에 매핑해 보여준다.

## 5. 키워드

해양, 해양수산, 항만, 선박, 항행, 수로, 전자해도, 해양정보, 해양데이터, 해양공간정보, VTS, 항로표지, 해양안전, 스마트항만, 자율운항, GIS, 공간정보, 지리정보시스템, 공간데이터, 데이터 표준화, 데이터 모델링, API, 정보시스템, 플랫폼, 시스템 구축, 시스템 고도화, 시각화, 모니터링, 관제, 디지털트윈, 클라우드, 빅데이터, 공공데이터, 국제표준, S-100

## 6. 데이터와 확장 구조

- 공통 공고 모델은 `NormalizedNotice`를 사용한다.
- 필수 필드는 `sourceId`, `externalId`, `title`, `url`, `publishedAt`, `deadlineAt`, `organization`, `budgetAmount`, `category`, `matchedKeywords`, `score`, `metadata`다.
- 출처별 API 호출과 정규화는 `NoticeSource` 어댑터로 분리한다.
- MVP 구현체는 공공데이터개방표준서비스 기반 `g2bSource` 하나만 만든다.
- 후속 구현체 또는 보강 서비스로 입찰공고정보서비스 기반 `g2bBidPublicDetailService`를 추가한다.
- 1차 수집 결과와 2차 상세 보강 결과는 `sourceId`, `externalId`, `bidNtceNo`, `bidNtceOrd`를 기준으로 매핑한다.
- 상세 보강 필드는 우선 `notices.metadata`에 저장하고, 첨부파일 링크가 중요해지면 `notice_attachments` 테이블을 추가한다.
- 후속 출처 추가 시 공통 웹/Slack/AI 파이프라인은 유지하고 새 어댑터만 추가한다.

## 7. 나라장터 API 역할 분담

| 구분 | API | 역할 |
|---|---|---|
| 1차 목록 수집 | 공공데이터개방표준서비스 `PubDataOpnStdService` | 넓게 입찰공고를 가져오고 키워드/점수 기준으로 후보를 만든다. |
| 2차 상세 보강 | 입찰공고정보서비스 `BidPublicInfoService` | 후보 공고만 대상으로 첨부파일, 상세 URL, 참가 조건, 기초금액 등 상세 정보를 보강한다. |

두 API는 같은 `G2B_SERVICE_KEY`를 사용하고, 호출 대상은 서비스별 Base URL로 구분한다. 1차 목록 수집은 `G2B_STANDARD_API_BASE_URL`, 2차 상세 보강은 `G2B_BID_PUBLIC_API_BASE_URL`을 사용한다.

2차 상세 보강은 API 호출량과 처리 비용을 줄이기 위해 점수 0점 공고에는 수행하지 않는다.

## 8. Slack 알림

Slack 메시지는 다음 정보를 포함한다.

- 공고명
- 수요기관 또는 공고기관
- 입찰 마감일
- 적합도 점수
- 매칭 키워드 기반 추천 이유
- 웹 상세 페이지 URL

낙찰방법이 `수의시담`인 공고는 후보 처리 단계에서 제외되므로 새 후보 Slack 묶음 알림에도 포함하지 않는다.

## 9. AI 전략 메모

첨부파일은 `kordoc`으로 Markdown 변환 후 LLM adapter에 전달한다. 전략 메모는 다음 형식으로 작성한다.

- 추천 이유
- 블루맵 역량 매핑
- 제안서에 녹일 전략
- 확인해야 할 리스크
- 다음 행동

LLM provider는 OpenAI 또는 Claude로 교체 가능해야 하며, API 키가 없으면 개발용 mock 응답을 제공한다.

## 10. 오류 처리

- 나라장터 API 오류는 수집 결과에 실패 사유를 남긴다.
- Slack Webhook이 없으면 알림은 건너뛰되 웹 수집은 계속 동작한다.
- PDF/HWPX/HWP 변환 실패 시 운영자가 이해할 수 있는 메시지를 보여준다.
- LLM 오류 시 첨부파일 변환 결과는 유지하고 전략 메모 실패 상태를 표시한다.
- 2차 상세 보강 API가 실패해도 1차 수집 공고는 유지하고, 상세 보강 실패 상태만 기록한다.

## 11. 테스트 기준

- 나라장터 샘플 응답을 `NormalizedNotice`로 변환한다.
- 키워드 매칭과 점수 계산이 동등 가중치로 동작한다.
- 저장된 공고는 중복 Slack 알림 대상이 되지 않는다.
- 낙찰방법이 `수의시담`인 공고는 후보 저장과 Slack 알림 대상에서 제외된다.
- Slack 알림의 웹 상세 URL이 정상 생성된다.
- PDF/HWPX/HWP 업로드, `kordoc` 변환, AI 전략 메모 생성 흐름을 검증한다.
- 후속 상세 보강 구현 시 1차 후보 공고와 입찰공고정보서비스 상세 응답이 동일 공고번호/차수로 매핑되는지 검증한다.
